# 技能树系统设计文档（完整可扩展版）

> 版本：v1.0 · 目标：将当前「每职业 5 个固定技能」的粗糙框架，升级为 **多层分支 + 解锁条件 + 可升级节点 + 洗点机制** 的完整技能树系统。
> 设计原则：**数据驱动、与现有架构解耦、预留动态调参接口**（与项目已有的 `BAL` 常量风格一致）。

---

## 0. 设计目标与现有架构适配

当前游戏的技能系统是一个扁平列表：

```js
const SKILL_DEFS = {
  warrior: [ {id:'atk',...}, {id:'fire',...}, {id:'cleave',...}, {id:'rage',...}, {id:'berserk',...} ],
  mage:    [...], taoist: [...]
};
```

问题：技能固定、无成长、无分支、无流派差异。本设计在不推翻 `SKILL_DEFS` 的前提下，在其之上叠加一层 **技能树（Skill Tree）**，二者职责分离：

| 现有系统 | 职责 | 技能树新增职责 |
|---|---|---|
| `SKILL_DEFS` | 技能「是什么」（id/图标/基础cd/基础系数/特效） | 技能「变强多少」（树节点增强其 `damageMult`/`cd`） |
| `calcPlayerStats` + `extraBonuses` | 汇总属性 | 树**被动节点**写入 `extraBonuses`（atk/def/hp/cdr…） |
| `Account.onLevelUp()` | 升级钩子 | 在此**发放技能点** |
| `saveGame()` | 持久化 | 增加 `skillTree` 状态字段 |

**集成路径（最小侵入）**：树被动 → `extraBonuses`；树主动增强 → `playerStats.skillMult[skillId]` / `skillCdr[skillId]`，由 `useSkill()` 读取。

## 1. 技能树结构

### 1.1 三主分支 × 六层递进

每个职业拥有 **3 个主分支**，每个分支 **6 层（Tier）**，共约 **18 个节点 / 职业**：

- **战斗（Combat）** —— 输出最大化：攻击%、技能增伤、暴击、连击、终极技。
- **生存（Survival）** —— 耐久与回复：生命%、防御%、吸血、减伤、再生、绝境爆发。
- **精通（Mastery）** —— 节奏与机制：移速、冷却缩减(CDR)、破甲、怒气/蓝量回收、群战、武器大师。

> 法师/道士分支名称做主题替换（战斗→奥术、生存→守护、精通→道术），**结构完全复用**，仅 `effects` 指向不同 `stat`/`skill`。

### 1.2 拓扑与解锁依赖

- **根节点统一**：进入任一分支持有 1 个免费「入门」节点（T1），无前置。
- **纵向递进**：同分支内 `T(n)` 解锁需 `T(n-1)` 已 ≥1 级（或满足 `prereq` 指定节点等级）。
- **横向弱耦合**：允许「战斗 T3」与「生存 T2」并行点，但 **终极节点（T6）需本分支 T5 满级 + 角色等级门槛**，制造「专精终点」。
- **跨分支协同**：设计上让不同分支的节点产生协同（如 战斗·烈火强化 × 精通·冷却掌控 = 高频烈火），鼓励混搭而不强制。

### 1.3 结构示意（以战士为例，纵向树状 + 分支并列）

```
                  [角色等级门槛 Lv.10]
        ┌────────────────┼────────────────┐
    ┌───战斗 Combat ───┐ ┌─生存 Survival ─┐ ┌─精通 Mastery ─┐
    │ T1 武器精通       │ │ T1 强健体魄    │ │ T1 疾风步     │
    │   +atk% (被动)    │ │   +Hp% (被动)  │ │   +spd% (被动)│
    ├──────────────────┤ ├───────────────┤ ├───────────────┤
    │ T2 烈火强化       │ │ T2 铁壁       │ │ T2 冷却掌控   │
    │   fire伤害+ (增)  │ │   +def/减伤   │ │   cdr% (被动) │
    ├──────────────────┤ ├───────────────┤ ├───────────────┤
    │ T3 横扫扩展       │ │ T3 嗜血       │ │ T3 怒气回收   │
    │   cleave范围+     │ │   吸血%       │ │   mp回复+     │
    ├──────────────────┤ ├───────────────┤ ├───────────────┤
    │ T4 狂暴延长       │ │ T4 不屈       │ │ T4 破甲       │
    │   berserk增强     │ │   低血减伤    │ │   armorPen%   │
    ├──────────────────┤ ├───────────────┤ ├───────────────┤
    │ T5 致命连击       │ │ T5 再生       │ │ T5 群战大师   │
    │   crit+/critDmg+  │ │   hpRegen     │ │   多目标增伤  │
    ├──────────────────┤ ├───────────────┤ ├───────────────┤
    │ T6 战神降临(终极) │ │ T6 绝境爆发   │ │ T6 武器大师   │
```

> UI 推荐布局：**三列纵向树状**（每列一个分支），列间留白，节点用连线表达层级；也支持「扇形展开 / 网格平铺」切换（见 §4）。

## 2. 技能节点属性（数据模型）

每个节点是一个完整的数据对象，字段如下：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 唯一标识，如 `war_combat_t3` |
| `branch` | enum | `combat` / `survival` / `mastery` |
| `tier` | int 1–6 | 所处层，决定解锁门槛与消耗基数 |
| `name` | string | 节点显示名 |
| `icon` | string | emoji 或图标资源 key |
| `type` | enum | `passive`（被动属性）/ `augment`（增强某技能）/ `active`（新主动技，仅 T6 终极） |
| `desc` | string | 基础描述（随等级动态拼接数值） |
| `baseCd` | float | 仅 `active` 节点：基础冷却（秒） |
| `duration` | float | 仅 `active`/`buff` 类：持续时间（秒） |
| `resource` | {type, cost} | 消耗类型：`mp` / `sp`(技能点，用于释放) / `none`；`cost` 数值 |
| `prereq` | string[] | 前置节点 id 列表（需全部 ≥1 级） |
| `reqLevel` | int | 角色等级门槛（通常 T5/T6 才需要） |
| `costPerLevel` | int[] | 每升 1 级消耗的**技能点**数组（长度=maxLevel），实现递增曲线 |
| `maxLevel` | int | 可升级等级上限（通常 1–5） |
| `effects` | Effect[] | 每级生效的数值，见下 |
| `balanceKey` | string | 指向 `SKILL_TREE_BAL` 的调参键（动态接口） |

`Effect` 结构：
```js
{ stat: 'atkPct' | 'hpPct' | 'cdr' | 'critRate' | 'skillMult' | ...,
  skill: 'fire' | null,        // 仅 augment 填，指定增强的技能 id
  perLevel: 0.04,               // 每级数值（百分比用小数，如 0.04 = +4%）
  isPercent: true }
```

### 2.1 节点示例（战士·战斗分支节选）

```js
{ id:'war_combat_t2', branch:'combat', tier:2, name:'烈火强化', icon:'🔥',
  type:'augment', desc:'强化烈火剑法，提升其伤害',
  resource:{type:'none'}, prereq:['war_combat_t1'], reqLevel:0,
  maxLevel:4, costPerLevel:[1,1,2,2],
  effects:[ { stat:'skillMult', skill:'fire', perLevel:0.08, isPercent:false } ],
  balanceKey:'augmentDmg' },

{ id:'war_combat_t6', branch:'combat', tier:6, name:'战神降临', icon:'⚡',
  type:'active', desc:'10秒内攻击+40%、吸血+15%',
  baseCd:90, duration:10, resource:{type:'mp', cost:80},
  prereq:['war_combat_t5'], reqLevel:30, maxLevel:1, costPerLevel:[6],
  effects:[ { stat:'atkPct', perLevel:0.40, isPercent:true },
            { stat:'lifesteal', perLevel:0.15, isPercent:true } ],
  balanceKey:'ultimate' }
```

## 3. 解锁与升级机制

### 3.1 技能点来源（SKILL_POINT 经济）

| 来源 | 规则 | 总量估算（Lv.1→30） |
|---|---|---|
| 升级奖励 | 每升 1 级 +1 点 | 29 点 |
| 里程碑 | 每 5 级额外 +2；每 10 级额外 +3 | (6×2)+(3×3)=21 点 |
| 主线任务 | 每完成 1 章 +1 点（共 12 章） | 12 点 |
| 隐藏成就 | 每个 +2 点（设计 6 个） | 12 点 |
| 职业起始 | 创建角色即 +2 | 2 点 |
| **合计** | | **≈ 76 点** |

> 设计意图：满级约 76 点，而**全节点点满需 ≈ 141 点**（3 分支 × 47，见 §5 表），故玩家只能覆盖 ~54% 节点 —— **强制取舍**，但任一分支可基本点满（单分支成本 ≈ 47 点），支撑「专精流」与「三修残次流」两种极端。

### 3.2 升级消耗曲线（递增）

每个节点的 `costPerLevel` 实现递增；推荐 **分层基数 × 等级** 的温和曲线：

```js
// SKILL_TREE_BAL.costBaseByTier = { 1:1, 2:1, 3:2, 4:2, 5:3, 6:4 }
// 节点升级到第 L 级花费 = costBaseByTier[tier] * L
// 例：T3 节点(maxLevel4) 总花费 = 2*1 + 2*2 + 2*3 + 2*4 = 20 点
```

可选指数曲线（PvP 服用）：`cost(L) = ceil(1.4^(L-1))` → 1,2,3,4… 前期便宜后期贵。默认用温和曲线，参数全在 `SKILL_TREE_BAL` 可调。

### 3.3 解锁条件判定

节点可解锁需**同时满足**：
1. 前置节点 `prereq` 全部已 ≥1 级；
2. 角色等级 ≥ `reqLevel`；
3. 拥有足够未分配技能点支付 `costPerLevel[当前等级]`；
4. 未达 `maxLevel`。

### 3.4 洗点 / 重置机制

- **入口**：技能树面板「重置」按钮。
- **代价**：消耗金币，递增曲线 `resetCost = round( SKILL_TREE_BAL.resetBase * (resetCount+1)^1.5 )`，`resetBase=500`。
- **效果**：返还该职业全部已投技能点、清空 `skillState.spent`、重置 `resets++`；被动/增强立即失效，重算属性。
- **保护**：重置后 24h 内不可再次重置（防刷点）；首充/新手前 3 天免费洗点 1 次（运营友好）。
- **不返还**：已消费于「升级里程碑」的技能点（来自任务/成就）不回收，仅返还自由分配部分——简化实现，避免负数。

### 4.1 视觉布局

**推荐：三列纵向树状（Vertical Tree）**，理由——与 §1.3 结构一一对应，层级清晰、依赖可见。备选 `fan`（扇形展开，适合触屏）/ `grid`（网格平铺，适合节点极多）。

### 4.2 节点状态与视觉反馈

| 状态 | 判定 | 视觉 |
|---|---|---|
| `locked` 锁定 | 前置未满足 / 等级不足 | 灰色 + 锁图标 🔒，连线虚线 |
| `unlockable` 可解锁 | 条件满足 + 有点数 | 彩色描边脉冲（呼吸动画），高亮可点 |
| `unlocked` 已点1级 | spent≥1 且 <max | 实心底色 + 图标亮起 |
| `upgraded` 已升级 | 1<spent<max | 外环显示进度（如 3/4 段亮） |
| `maxed` 满级 | spent==max | 金色光晕 + ✦ 角标，呼吸金光 |

### 4.3 悬停 / 点击信息展示

- **悬停（Hover）ToolTip**：
  ```

  需求等级：无
  ```
- **点击（Click）→ 详情面板**：
  - 当前等级 / 最大等级
  - 当前效果数值 → 下一级数值（箭头对比）
  - 按钮：`[升级]（消耗 N 点）` `[重置此节点]（返还 M 点）`
  - 若 `type==active`：额外显示 `CD / 持续 / 释放消耗`

## 5. 平衡性参数

### 5.1 分支效果上限（防碾压）

通过 `SKILL_TREE_BAL.branchCap` 限制单分支总贡献，确保无分支一家独大：

| 分支 | 核心属性 | 单分支满点总上限 | 设计意图 |
|---|---|---|---|
| 战斗 | atk%/skillMult/crit | atk +30% 或 指定技能 +40%（二选一路径） | 纯输出但脆 |
| 生存 | hp%/def%/dmgReduce/lifesteal | hp +45%、dmgReduce +25%、吸血 +12% | 肉但输出平庸 |
| 精通 | cdr/破甲/移速/mp回复 | cdr +18%（封顶）、破甲 +20% | 节奏流，靠协同放大其他分支 |

> **协同红利**：跨分支组合产生超线性收益（如 战斗·致命连击 crit+ × 精通·群战 → 暴击群伤），但**单分支内部**严格封顶，避免「只点战斗就无敌」。

### 5.2 数值平衡参考表（战士·战斗分支节选，全职业同构）

| 节点 | 分支 | 类型 | maxLv | 每级消耗 | 每级效果 | 满级总效果 | 平衡备注 |
|---|---|---|---|---|---|---|---|
| 武器精通 T1 | 战斗 | 被动 | 5 | 1,1,2,2,3 | atk +3% | atk +15% | 基础输出基底 |
| 烈火强化 T2 | 战斗 | 增 | 4 | 1,1,2,2 | fire伤害 +8% | +32% | 需配合火技能 |
| 横扫扩展 T3 | 战斗 | 增 | 3 | 2,2,3 | cleave范围 +10% | +30%范围 | 群战向 |
| 狂暴延长 T4 | 战斗 | 增 | 3 | 2,2,3 | berserk时长 +20% | +60%时长 | 爆发流 |
| 致命连击 T5 | 战斗 | 被动 | 4 | 2,3,3,4 | critRate+2%/critDmg+5% | +8%暴击/+20%暴伤 | 后期收益高 |
| 战神降临 T6 | 战斗 | 主动 | 1 | 6 | atk+40%/吸血+15% (10s) | 终极技 | reqLv30, CD90s |

> 单分支满点成本 ≈ 1+1+2+2+3 + 1+1+2+2 + 2+2+3 + 2+2+3 + 2+3+3+4 + 6 = **40 点**，与 §3.1 总 76 点吻合（可点满 1 主分支 + 另两分支各半）。

### 5.3 技能点经济平衡表

| 项目 | 数值 | 说明 |
|---|---|---|
| 满级可获总点 | 76 | 来源见 §3.1 |
| 全节点满点成本 | ~141 | 强制取舍 |
| 单分支满点成本 | ~47 | 支持「专精流」 |
| 三修残次覆盖率 | ~54% | 平衡不鼓励全修 |
| 重置基础费 | 500 金 ×(n+1)^1.5 | 第1次≈500，第2次≈1375 |

### 5.4 动态调参接口（SKILL_TREE_BAL）

所有可调参数集中在单一对象，**改这里即全局生效**，无需动节点数据：

```js
const SKILL_TREE_BAL = {
  pointsPerLevel: 1,
  milestone: { every5:+2, every10:+3 },
  questPoint: 1, achievementPoint: 2, startBonus: 2,
  costBaseByTier: { 1:1, 2:1, 3:2, 4:2, 5:3, 6:4 },   // 升级消耗曲线
  costCurve: 'linear',                                  // 'linear' | 'exp'
  branchCap: { combat:{atkPct:0.30}, survival:{hpPct:0.45,dmgReduce:0.25}, mastery:{cdr:0.18} },
  augmentDmg: 1.0,        // 全局增伤系数（热修用）
  ultimate: 1.0,          // 终极技系数
  resetBase: 500,         // 洗点基础费
  resetExp: 1.5,         // 洗点费用指数
  resetCooldownH: 24,     // 洗点冷却（小时）
};
```

## 7. 解锁逻辑伪代码

```js
// 角色升级时发放技能点（挂入 Account.onLevelUp / 升级循环）
function awardSkillPoints(level) {
  let gained = SKILL_TREE_BAL.pointsPerLevel;
  if (level % 5 === 0) gained += SKILL_TREE_BAL.milestone.every5;
  if (level % 10 === 0) gained += SKILL_TREE_BAL.milestone.every10;
  skillState.points += gained;
}

// 查询节点当前状态
function getNodeStatus(nodeId) {
  const node = SKILL_TREE_DEFS[playerProfession].nodes[nodeId];
  const lv = skillState.spent[nodeId] || 0;
  if (lv >= node.maxLevel) return 'maxed';
  if (lv > 0) return lv === 1 ? 'unlocked' : 'upgraded';
  if (canUnlock(nodeId)) return 'unlockable';
  return 'locked';
}

// 能否解锁/升级（核心判定）
function canUpgrade(nodeId) {
  const node = SKILL_TREE_DEFS[playerProfession].nodes[nodeId];
  const lv = skillState.spent[nodeId] || 0;
  if (lv >= node.maxLevel) return false;                       // 满级
  for (const p of node.prereq)                                 // 前置
    if ((skillState.spent[p]||0) < 1) return false;
  if (playerLevel < node.reqLevel) return false;               // 等级
  const cost = node.costPerLevel[lv];                          // 点数
  if (skillState.points < cost) return false;
  return true;
}

// 执行升级
function upgradeNode(nodeId) {
  if (!canUpgrade(nodeId)) { Sound.deny(); return false; }
  const node = SKILL_TREE_DEFS[playerProfession].nodes[nodeId];
  const lv = skillState.spent[nodeId] || 0;
  skillState.points -= node.costPerLevel[lv];
  skillState.spent[nodeId] = lv + 1;
  recomputeBonuses();
  Sound.levelup();
  saveGame();
  return true;
}

// 洗点（重置单职业全部节点）
function resetTree() {
  if (skillState.resets > 0 && now - lastReset < SKILL_TREE_BAL.resetCooldownH*3600e3) return false;
  const cost = Math.round(SKILL_TREE_BAL.resetBase * Math.pow(skillState.resets+1, SKILL_TREE_BAL.resetExp));
  if (playerGold < cost) { Sound.deny(); return false; }
  playerGold -= cost;
  let refunded = 0;
  for (const id in skillState.spent) {
    const node = SKILL_TREE_DEFS[playerProfession].nodes[id];
    for (let i=0;i<skillState.spent[id];i++) refunded += node.costPerLevel[i];
  }
  skillState.points += refunded;
  skillState.spent = {};
  skillState.resets++;
  recomputeBonuses();
  saveGame();
  return true;
}

// 将已点节点折算进属性（写入 extraBonuses / skillMult）
function recomputeBonuses() {
  const bonuses = {};
  const skillMult = {};      // { skillId: multiplierAdd }
  const nodeDefs = SKILL_TREE_DEFS[playerProfession].nodes;
  for (const id in skillState.spent) {
    const lv = skillState.spent[id];
    const node = nodeDefs[id];
    const scale = SKILL_TREE_BAL[node.balanceKey] ?? 1;
    for (const eff of node.effects) {
      const v = eff.perLevel * lv * scale;
      if (eff.skill) skillMult[eff.skill] = (skillMult[eff.skill]||0) + v;
      else bonuses[eff.stat] = (bonuses[eff.stat]||0) + v;
    }
  }
  extraBonuses = bonuses;                 // calcPlayerStats 已消费
  playerStats.skillMult = skillMult;      // useSkill 读取
  playerStats = calcPlayerStats(playerProfession, playerLevel, Object.values(playerEquipment));
}
```

> `useSkill()` 改造（一行）：计算伤害时 `const mult = skill.damageMult + (playerStats.skillMult?.[skill.id]||0);`

## 9. 与小游戏 / 多端适配

- **数据驱动**：`SKILL_TREE_DEFS` / `SKILL_TREE_BAL` 是纯 JSON 友好对象，H5 与小游戏**共用同一份**。
