## 0. 设计总览（Executive Summary）

《传奇·觉醒》是一款面向移动端 + PC 浏览器的 **2D 像素动作 ARPG**。它继承经典传奇"打宝—成长"内核，但用现代 H5 技术栈（单文件 Canvas2D + 零依赖 Node 后端）实现，可在浏览器直运行、可转微信小游戏。

本作的核心乐趣来自一条铁三角闭环：

```
  刷怪/打 Boss ──掉落随机词条装备──► 强化/附魔/交易 ──► 战力提升
        ▲                                                              │
        └──────────── 挑战更高阶地图与多阶段 Boss ◄───────────────────┘
```

设计目标是：**10 分钟上手，1000 小时仍可追求极品**。所有系统围绕"随机惊喜 + 可控成长 + 社交交易"展开。

| # | 支柱 | 一句话定义 | 设计含义 |

## 2. 世界观与叙事（背景 / 剧情 / 主线）

> 版权声明：本设计参考"双界穿越 / 修炼系统 / 打怪爆装 / 药水传送卷 / 无敌天下"的**题材内核**，为**原创世界观与剧情**，不复制任何小说具体情节、人名、地名与文本。

### 2.1 背景与世界观（原创）

**双界设定**
- **沧澜界（现实）**：科技与古文明并存的大陆，近年频现"蚀光"异象，古老遗迹苏醒。
- **渊墟（灵界）**：与沧澜通过"裂隙"相连的修炼之境，由上古"裂界之战"崩裂为 **五块浮空群系**。渊墟正被"蚀潮"侵蚀，兽人、亡灵、元素、冰族、龙裔等怪物涌出。

**核心设定（题材内核，非抄袭）**
- 主角本是沧澜的遗迹调查员，在古墓中触发 **"觉醒印记"**，获得 **「渊墟修炼系统」**——可将意识潜入渊墟，通过战斗获取经验、爆出装备、使用药水与 **随机传送卷** 在群系间穿梭。
- 系统不靠外力施舍，全凭"打怪—爆装—变强"的循环，最终目标是集齐散落的 **英灵核心**、封印裂隙、唤醒渊墟——即"于乱世中修炼至无敌"的原创表达。

**五群系 = 五幕舞台（与现有 5 张地图一一映射）**
| 群系 | 对应地图 | 生态/威胁 |
|------|---------|----------|
| 启明原野 | 比奇城郊 | 新手庇护所，兽潮频发 |
| 枯骨回廊 | 僵尸洞窟 | 古代文明遗迹，亡灵盘踞 |
| 焚心熔狱 | 烈焰火山 | 岩浆与火元素，炎之核心守者 |
| 霜寂冰原 | 冰封雪原 | 冰族与雪原部族 |
| 渊龙深渊 | 魔龙深渊 | 最终战场，远古龙裔 |

### 2.2 剧情 / 故事（五幕，多阶段 Boss = 高潮节点）

- **序章·觉醒印记**：沧澜突发蚀光，主角入古墓触印，初临启明原野，学会战斗、拾取、用药与传送。
- **第一幕·原野告急**：兽潮围村，主角击退兽群、救下守村人，得知"英灵核心"碎落五界，唯集齐可封印裂隙。
- **第二幕·枯骨低语**：深入回廊遗迹，初遇 **尸王**（双阶段）——第一阶段是腐化的守墓将，第二阶段唤醒远古战魂，主角夺得第一块核心碎片与碑文线索。→ *对应职业进阶一（Lv10）觉醒*
- **第三幕·焚心试炼**：熔狱之中 **火焰魔君**（三阶段）镇守炎之核心；主角经历"灼烧—狂暴—真身"三阶，习会元素试炼，取核心。
- **第四幕·霜原誓约**：冰原 **冰霜巨魔**（双阶段）守霜之核心；主角与雪原部族结盟，以誓约之力破冰封。→ *对应职业进阶二（Lv22）觉醒*
- **第五幕·深渊终焉**：集齐核心深入渊龙深渊，决战 **远古巨龙**（三阶段：幼龙—蚀龙—真龙），封印裂隙、唤醒渊墟，主角于乱世中成就"无敌"。

> 每个 Boss 的阶段切换即剧情转折 + 主角实力跃升的仪式感，复用现有多阶段机制（`BOSS_DATA`）。

### 2.5 主线任务系统架构（可落地规范）

> 本节把 §2.3 的"任务线"升级为**可实现的任务系统规范**：任务链结构、触发、进度、奖励、状态机、NPC 节点、依赖、难度曲线、失败/重试，并给出与现有 `profile` 字段（`mainQuests` / `clearedMaps` / `advancement`）对接的数据模型。配套数据见 `quest_defs.json`。

#### 2.5.1 任务链结构（三层：幕 → 章节 → 目标）

| 层级 | 含义 | 数量 | 与现有代码映射 |
|------|------|------|---------------|
| **幕 Arc** | 对应一张地图 + 一个多阶段 Boss | 6（序 + 五幕） | `MAP_DEFS[i]` + `BOSS_DATA` |
| **章节 Chapter** | 幕内剧情段落，含 1–4 个目标 | 每幕 1–3 | 运行时 `mainQuests` 键 |
| **目标 Objective** | 最小可完成单元 | 每章 1–4 | 进度计数器 `questObjectives` |

**目标类型枚举（ObjectiveType）**
- `reach` 抵达指定地图/坐标
- `kill` 击杀指定怪物 N 只（可被 `onKill` 事件累加）
- `collect` 收集指定掉落物 N 个（绑定 `equipment_affix_db` 掉落）
- `talk` 与指定 NPC 对话（消费对话节点）
- `boss` 击败指定 Boss（含阶段要求）
- `escort` 护送 NPC 至坐标（可含中途受击判定）
- `explore` 探索/扫描指定区域（进入触发区即完成）

主线为**线性**（幕 i 全完成才解锁幕 i+1）；每幕附 1 条**可选支线**（不给核心碎片，给金币/词条/技能点，用于留存与追赶）。

#### 2.5.2 任务触发条件（Trigger）

| 触发类型 | 说明 | 时机 |
|---------|------|------|
| `auto` | 进入地图/登录自动接取 | 满足条件即刻 |
| `level` | 达到等级门槛 | 升级时检查 |
| `npc` | 与 NPC 对话接取 | 玩家主动 |
| `event` | 特定事件（击杀 Boss、拾取核心） | 事件总线 |
| `item` | 持有/使用道具（如传送卷） | 背包变更 |

- **前置依赖（prereq）**：章节可声明 `prereqArc` / `prereqChapter`，未满足则状态为 `locked`。
- **互斥**：主线不互斥；支线可声明 `excludes`（避免同时接取冲突支线）。
- **去重**：同一目标只计数一次（`questObjectives[id].cur` 封顶 `target`）。

#### 2.5.3 任务进度追踪（Progress Tracking）

- 运行时维护 `profile.questObjectives: { [objectiveId]: { cur, target } }`。
- 全局**事件总线** `emit(event, payload)` 驱动计数：`kill`→`onKill(monsterId)`、`pickup`→`onPickup(itemId)`、`enterMap`→`onEnterMap(mapId)`、`talk`→`onTalk(npcId)`、`bossDown`→`onBossDown(bossId,phase)`。
- 每个目标完成 → `cur>=target` 标记该目标 `done`；章节内全部目标 `done` → 触发章节奖励 + `onComplete`。
- **UI 追踪栏**：HUD 右侧显示"当前章节名 + 下一个未完成目标 + 进度百分比"；点击展开全目标列表。
- **持久化**：每次目标进度变化即写入 `profile`（随 `saveGame`/`syncProfile` 落盘与后端同步，复用 §12.3 数据互通）。

#### 2.5.4 任务奖励体系（Rewards）

| 奖励档 | 触发 | 内容 |
|--------|------|------|
| 目标即时奖励 | 单目标完成 | 小经验 / 少量金币（正反馈） |
| 章节完成奖励 | 章节全目标完成 | 经验 + 金币 + 技能点(1) + 可能掉装备词条 |
| 幕完成奖励 | 全幕完成 | **核心碎片（剧情解锁）** + 进阶点或属性点 + 地图传送解锁 |

- **首通 vs 重复**：主线核心碎片/剧情只首通给予；击杀/收集类目标可重复刷材料但不重复给碎片。
- **奖励与系统联动**：核心碎片累计是终章解锁条件；`reach` 解锁地图传送门槛（`minLevel`）；`boss` 完成驱动进阶门槛（见 §2.5.7）。

#### 2.5.5 任务状态管理（State Machine）

```
        prereq满足
  locked ──────────► available
   ▲                   │ 接取(auto/npc)
   │                   ▼
   │                active ──┐
   │                 │      │ 全部目标 done
   │                 │      ▼
   │              abandoned  completed ──► (解锁下游)
   │                 ▲           │
   └─────────────────┘           │ 主线允许"重置当前章"
                                 ▼
                              failed(仅战斗类可触发→原地复活重试)
```

| 状态 | 含义 | 玩家可见行为 |
|------|------|------------|
| `locked` | 前置未满足 | 任务列表灰显 |
| `available` | 可接未接 | 列表高亮"新" |
| `active` | 进行中 | 追踪栏显示 |
| `completed` | 已完成 | 打勾，可回顾 |
| `abandoned` | 已放弃 | 仅支线可放弃，主线禁用 |
| `failed` | 已失败 | 主线不进入此态（防卡死） |

- **主线不可真正放弃**：`abandoned` 仅对支线开放；主线提供「重置本章」回到 `active` 首目标，避免进度卡死。
- **状态存 `profile.mainQuests: { [chapterId]: state }`**，与 §2.5.3 进度同步落盘。

#### 2.5.6 关键 NPC 交互节点

每幕设 1 个**核心 NPC**（剧情锚点），交互遵循：`idle → talking → (giver|reward) → done` 状态机。

| 幕 | NPC | 角色 | 交互节点 |
|----|-----|------|---------|

| 二 | 碑灵·残魂 | 线索者 | 枯骨低语 → 揭示尸王真相 → 战后授核心碎片② |
| 三 | 炎心·炽 | 试炼者 | 焚心试炼 → 三阶考验 → 授炎之核心③ |
| 四 | 雪原长老·霜 | 结盟者 | 霜原誓约 → 部族结盟 → 授霜之核心④ |
| 五 | 龙裔先知·渊 | 终局者 | 深渊终焉 → 集齐核心 → 决战引导 → 封印裂隙 |

NPC 对话树用 `nodes: { id, text, options:[{label, next|action}] }` 表达，`action` 可触发 `giveQuest` / `completeObjective` / `reward`。

#### 2.5.7 任务依赖关系（DAG）

主线依赖为**有向无环图**，关键边：

```
序章 ──► 第一幕 ──► [进阶一@Lv10] ──► 第二幕 ──► 第三幕 ──► [进阶二@Lv22] ──► 第四幕 ──► 第五幕
   │                                                     │
   └──────── 核心碎片①②③④ 累计 = 第五幕解锁条件 ─────────┘
```

- **跨系统依赖**：进阶一（Lv10）是第二幕进入门槛；进阶二（Lv22）是第四幕门槛（复用 `ADVANCEMENTS`）。
- **核心碎片门控**：`profile.coreFragments` 收集 4 块后，第五幕 `boss` 目标前的 `reach` 深渊传送才解锁。
- 依赖检查函数 `canStart(chapterId)` 在 `auto`/`npc` 触发时统一调用。

#### 2.5.8 难度曲线（Difficulty Curve）

| 幕 | 等级门槛 | 目标数(典型) | 主要威胁 | 难度系数 |
|----|---------|------------|---------|---------|
| 序 | 1 | 4（教学） | 无战斗/野狼×3 | 0.4 |
| 一 | 1→5 | 4 | 兽潮(小怪群) | 0.7 |
| 二 | 5→10 | 5 | 尸王(双阶段) | 1.0 |
| 三 | 10→16 | 5 | 火焰魔君(三阶段) | 1.4 |
| 四 | 16→22 | 5 | 冰霜巨魔(双阶段)+部族 | 1.7 |
| 五 | 22→终 | 4 | 远古巨龙(三阶段) | 2.2 |

- 难度系数 = 怪物等级权重 × 目标密度 × Boss 阶段数；用于自动校准推荐装备品质与掉落保底（见 §6/§7）。
- 曲线刻意"阶梯式"：每幕末 Boss 为峰值，幕间有等级/进阶缓冲，避免陡坡劝退。

#### 2.5.9 失败处理与重试（Failure & Retry）

- **主线无硬失败**：不存在 `failed` 态；死亡走既有"原地复活"惩罚（扣金币/掉少量耐久，不丢进度）。
- **目标重试**：`boss` 目标可无限重复挑战（Boss 阶段状态每战重置）；`collect` 漏收可回地图补刷；`escort` 失败仅重置该章节护送段。
- **防卡死三机制**：① 主线禁 `abandoned`；② 卡住可「重置本章」回到首目标；③ `auto` 触发在重新进入地图/登录时复检，防止漏接。
- **重试成本**：仅时间成本 + 复活惩罚，不惩罚叙事进度，符合 §10 留存设计。

## 4. 战斗系统与职业体系

### 4.1 战斗模型

- **实时动作 ARPG**：点击/摇杆移动，技能键释放，无回合等待。
- **操作**：PC = WASD 移动 + 1~6 技能 + 空格闪避；移动端 = 虚拟摇杆 + 技能按钮。
- **索敌**：技能可指向（鼠标/摇杆方向）或自动锁定最近目标。
- **反馈**（落实 P1）：命中触发 **hit-stop 顿帧（40-80ms）+ 屏幕震动 + 白光闪 + 伤害数字飘字**；暴击用更大字号 + 橙色；击杀有碎裂粒子。

### 4.2 属性体系

| 属性 | 缩写 | 说明 |
|------|------|------|
| 生命 | HP | 受击归零即死亡（可原地复活） |
| 法力 | MP | 释放技能消耗 |
| 主攻击 | PATK/MATK/TATK | 按职业区分：物理/魔法/道术 |
| 物理防御 | PDEF | 减免物理伤害 |
| 魔法防御 | MDEF | 减免魔法伤害 |
| 暴击率 | CRIT | 触发暴击的概率 (%) |
| 暴击伤害 | CDA | 暴击时的额外伤害 (%) |
| 攻击速度 | ASPD | 影响普攻与技能冷却 |
| 命中 | HIT | 对抗闪避 |
| 闪避 | EVA | 概率免伤 |
| 吸血 | LS | 造成伤害按比例回血 |
| 移速 | MOVE | 移动速度 |

### 4.3 伤害公式

```
主属性攻击 = 职业主攻击 × 技能系数
减伤率   = 防御 / (防御 + 防御常数K)      // K = 200 + 角色等级×8
基础伤害 = 主属性攻击 × 技能系数 × (1 - 减伤率)
浮动     = rand(0.95, 1.05)
暴击判定 = rand(0,100) < CRIT  →  最终伤害 × (1 + CDA/100)
吸血回血 = 最终伤害 × LS/100
最终伤害 = 基础伤害 × 浮动 × (暴击 ? 1+CDA/100 : 1)
```

> 设计意图：防御采用"常数分母"模型，保证高防御仍可被穿透（避免堆防无敌），符合 P2 装备驱动但需平衡。

### 4.8 战斗演出与反馈实现（已落地）

> 落实 P1「爽快的战斗」支柱：每一次出手都要有"打到了"的明确反馈。以下均已接入 `index.html` 实时战斗循环。

- **屏幕震动 `screenShake`**：命中/受击按强度叠加最大震幅并衰减，与 hit-stop 同帧触发，形成"冲击—定格—回弹"节奏。

| 吸血 + | 1%~5% | 低 |

**后缀（防御·功能）**

| 词条 | 数值区间 | 权重 |
|------|---------|------|
| 生命 + | 20~200 | 高 |
| 物理/魔法防御 + | 10~80 | 高 |
| 闪避 + | 1%~5% | 中 |

### 6.4 随机生成算法

```
function rollAffixes(itemLevel, quality):
    count = quality.affixCount
    result = []
    pool = shuffle(AFFIX_POOL weighted by itemLevel)
    usedNames = set()
    while result.length < count:
        affix = pool.pop()
        if affix.name in usedNames: continue          // 同名去重
        if conflicts(affix, result): continue          // 互斥检查
        range = affix.range × quality.rollMult × levelScale(itemLevel)
        value = rand(range.min, range.max) rounded
        result.push({ name, value, tier: rollTier(value, range) })
    return result
```

- `levelScale`：随地图等级提升区间（深渊图出的装备天然更高）。
- `rollTier`：单条词缀内部再分"普通/优良/完美"三档，决定该条数值在自身区间的位置，制造"这条极品、那条拉胯"的对比乐趣（落实 P3）。

### 6.5 掉落与装备等级

- 怪物/Boss 掉落时按 **所在地图 `itemLevel`** 决定装备底材等级。
- Boss 提升品质概率与固定词条数量（见 §6.1）。
- 已实现：词条掉落、品质分档音效（普通叮→传说嗡→神话全服通告）。

### 6.6 配置驱动规则引擎（实现：equip-rules.json + equip-rules.js）

> 用户需求 #3（规则联动）、#4（配置驱动/热更新）、#5（边界处理）的落地实现。所有词条池、品质参数、概率权重、评分规则均由 `equip-rules.json` 单一配置文件驱动，代码层 `equip-rules.js` 为纯逻辑 UMD 引擎（浏览器 `window.EquipRules` / Node `require` 双模式），**无需改代码即可调整规则**。

#### 6.6.1 文件结构与加载流程

| 文件 | 角色 | 大小/版本 |
|------|------|----------|
| `equip-rules.json` | 规则数据源（词条池 / 品质 / 权重 / 评分 / 升阶） | v2，47 词条 / 6 品质 |
| `equip-rules.js` | 规则引擎（生成/评分/评定/升阶/校验），UMD 双模式 | 内置 `DEFAULT_CONFIG` 离线兜底 |

加载链路（游戏启动）：
```
index.html init()
  ├─ applyConfigToGlobals()                      // 用当前 EquipRules.config 重建 QUALITY / QUALITY_LIST / AFFIX_DB 映射
  └─ EquipRules.loadFromURL('/equip-rules.json') // 拉取线上配置 → init(cfg)
         .then(applyConfigToGlobals)             // 配置生效后再同步全局
```
- **热更新**：运行时调用 `window.__game.reloadRules()` → `EquipRules.reload()`（复用 `lastUrl`）→ 重新拉取并 `applyConfigToGlobals()`，无需刷新页面。
- **离线兜底**：`equip-rules.js` 加载即用 `DEFAULT_CONFIG` 初始化；`loadFromURL` 失败仅告警，保持旧配置，绝不崩溃。
- **后端**：`server/index.js` 新增 `GET /api/equip-rules`（读取 `equip-rules.json` 返回），同时静态托管 `equip-rules.json`，双通道可用。

#### 6.6.2 词条三层稀有度与评分权重

词条按 `tier` 分三级，决定基础评分权重 `score`：

| tier | 含义 | 评分权重 `score` | 示例 |
|------|------|-----------------|------|
| basic | 基础属性词条（力量/生命/防御…） | 低（约 5~12） | A01 力量 + |
| rare | 稀有词条（战斗特效/特殊数值） | 中（约 20~40） | B/C/D 系列 |
| legendary | 传奇词条（特殊机制，如复活/反伤/穿透） | 高（约 50~90 + 特殊加成） | F01~F04 |

#### 6.6.3 品质 → 词条层概率联动（qualityAffixTierWeights）

每档品质配一组三层权重，roll 词条时先按该权重选层、再在层内按 `weight` 选具体词条。**高品质显著拉高高稀有度词条概率**：

| 品质 | basic | rare | legendary |
|------|-------|------|-----------|
| common | 100 | 0 | 0 |
| good | 80 | 20 | 0 |
| rare | 55 | 40 | 5 |
| epic | 35 | 50 | 15 |
| legendary | 15 | 55 | 30 |
| mythic | 0 | 35 | 65 |

> 这直接满足"高品质→高稀有度词条概率提升"。数值集中在 `equip-rules.json` 的 `qualityAffixTierWeights`，策划可随时调。

#### 6.6.4 评分算法与动态品质评定

**单词条分**：
```
tierScore = affix.tier 对应的基础分
norm      = (value - rawMin) / (rawMax - rawMin)        // 该条在其区间内归一到 0~1
single    = tierScore × (0.4 + valueWeight × norm)      // valueWeight 由 scoring.valueWeightNorm 控制
special 类额外 + scoring.specialBonus
```
**总评分** = 所有词条 `single` 之和。`generateEquipment` 输出携带 `score` 字段。

**动态品质评定** `judgeQuality(rolledQuality, score, allowDowngrade)`：
- 取 `qualityScoreThresholds`：common 0 / good 25 / rare 60 / epic 110 / legendary 200 / mythic 320。
- 计算 `jt` = 评分应处档位，`rt` = 掉落 roll 出的档位。
- 生成时 `allowDowngrade=false`：**只升不降**（评分超阈值则晋级到对应品质，如 rare 评分≥200 → 晋级 legendary），制造"欧皇翻盘"惊喜（探针实测 rare 有约 15%~20% 被晋级）。
- 替换词条时 `allowDowngrade=true`：评分跌破当前档阈值则**允许降级一档**，保证词条与品质始终自洽。

实测随机生成 5000 件，平均评分随品质单调递增：common≈7 / good≈16 / rare≈60 / epic≈109 / legendary≈258 / mythic≈387。

#### 6.6.5 生成管线（generateEquip 委托链）

```
generateEquipment(sourceLevel, prof, opts)            // index.html 入口，兼容旧调用（slotKey/affixLocks 透传）
   └─ EquipRules.generateEquip({qualityOverride|roll, sourceLevel, prof})
        1. rollQuality() 或 qualityOverride            // 先定品质
        2. rollAffixes(quality, sourceLevel, prof)      // 按 qualityAffixTierWeights 选层 → 层内按 weight 选词条
             · 写入 blocked[attr] + 遍历 conflicts 做互斥检测，同 attr 不重复
             · prof 过滤（职业专属词条）
             · 溢出保护：尝试上限 count*20+50 次，池耗尽优雅停止（日志提示）
        3. computeValue(affix, quality, sourceLevel)    // value = roll(min,max) × 品质倍率(qualityScale) × 等级缩放(levelScale)
             · kind=flat 显示 "+N"；kind=pct 显示 "+X%"；kind=special 显示名称
        4. toAffixOut() → 统一输出 {id,tier,attr,name,kind,type,displayValue,display,score,...}
        5. scoreAffixes() → 总评分
        6. judgeQuality(rolled, score, false) → 最终 quality（动态晋级）
```
消费契约（供 `calcPlayerStats` / 背包渲染）：只读 `af.type` + `af.displayValue`（数值字符串），`kind` ∈ {flat,pct,special}；背包渲染用 `af.display`。

#### 6.6.6 锁定 / 替换机制

- `lockAffix(equip, index)`：切换某词条锁定态；锁定的词条在重铸/替换/升阶加词时**始终保留**。
- `replaceAffix(equip, index, opts)`：基于现有词条构建可用池（排除自身 + blocked 同 attr + conflicts + 职业不符），加权随机重选；重算 `score` 并 `judgeQuality(..., true)` 重评定（允许降级）。池为空时安全失败，保留原词条并记日志。
- 重铸系统 `rollAffixesForReforge` 复用 `replaceAffix`，锁定词条不被洗掉。

#### 6.6.7 升阶系统（材料消耗 + 成功率）

升阶把装备提升一档品质，并可能追加词条 / 重算数值：

| 参数（equip-rules.json → upgrade） | 值 | 说明 |
|-----------------------------------|-----|------|
| materials | iron_ore:2, monster_core:1 | 基础消耗 |
| successBase / decay / min / max | 0.65 / 0.10 / 0.10 / 0.95 | 成功率 = clamp(successBase − decay×当前档序, min, max) |
| 材料消耗 | materials × (tier+1) | 越往上越贵 |

- `canUpgrade(equip)`：存在下一档品质才允许。
- `upgradeCost(equip)`：返回本次升阶材料清单。
- `upgradeSuccessRate(equip)`：返回本次成功率（已 clamp）。
- `upgradeEquip(equip, materials)`：校验材料充足 → 扣材料 → `Math.random()<rate` 决定成败；成功则 `quality` 提升 + 词条数值重 roll（含品质/等级缩放）+ 按新品质上限可能追加一条词条；失败则品质不变（材料已扣，符合"沉没成本"设计）。

#### 6.6.8 边界处理与参数校验

引擎 `validate(cfg)` 全量校验，覆盖：
- 品质缺失 / 未知品质引用 / `qualityScoreThresholds` 与 `qualities` 不一致 → 返回 `ok:false` 并**保留旧配置**不崩溃。
- `rollAffixes` 溢出保护：尝试次数上限 `count*20+50`，池耗尽立即停止（日志 `可用词条池耗尽`）。
- `affixMax > 词池总数`：按池大小封顶，不死循环。
- 空词条池：`generateEquip` 返回 0 词条而不报错。
- 词条层权重全 0：回退到 `rollQuality` 默认 common。
- 日志：`log(level,msg)` 环形缓冲 200 条 + `console` 输出，所有异常路径可观测（满足"完整错误日志输出"）。

#### 6.6.9 扩展指南（数据驱动，零代码改动）

调整任何规则只需改 `equip-rules.json` 并触发热加载（`__game.reloadRules()` 或刷新）：
- 调掉率 → 改 `qualities[*].dropWeight`
- 调某品质"多高稀" → 改 `qualityAffixTierWeights`
- 加新词条 → 在 `affixes` 追加一条（填 id/tier/attr/kind/min/max/weight/score/conflicts?/prof?）
- 调评分曲线 → 改 `scoring`（specialBonus / valueWeightNorm）或各 tier 的 `score`
- 调品质阈值 → 改 `qualityScoreThresholds`
- 调升阶 → 改 `upgrade`（materials / successBase / decay / min / max）

## 7. 装备品质与词条组合规则

### 7.1 品质分级

| 品质(key) | 颜色 | 随机词条数(实现) | 区间倍率 | 基础掉率(小怪池) |
|----------|------|----------------|---------|-------------------|
| 普通 common | 白 #aaaaaa | 1 | 1.0× | 60% |
| 优秀 good | 绿 #44ff00 | 1~2 | 1.1× | 25% |
| 稀有 rare | 蓝 #00aaff | 2~3 | 1.25× | 10% |
| 史诗 epic | 紫 #aa00ff | 2~3 | 1.5× | 3.8% |
| 传说 legendary | 橙 #ff8800 | 3~4 | 1.9× | 1% |
| 神话 mythic | 红 #ff0044 | 3~4 | 2.5× | 0.2% |

> 掉率由 `QUALITY.dropWeight` 归一化（总权 1000：600/250/100/38/10/2）；Boss 用各自 `dropQualXxx` 覆盖小怪池（见 `BOSS_DATA`）。"固定特权词条"列为 §7 前瞻规划项，**当前实现仅随机词条**（数量区间见 `affixMin/affixMax`）。区间倍率 `qualityScale` 叠加到每条词缀 roll 范围。

> 神话装备拥有 **唯一命名**（如"屠龙·神话"）与全服通告，是 P3/P5 的终极炫耀物。

### 7.2 品质 → 词条数量/区间映射

- 词条数严格按上表；区间倍率叠加到每条词缀的 roll 范围。
- 品质在掉落瞬间由 **独立一次 roll** 决定（先定品质，再按品质 roll 词条），保证"高品质必多词、且数值更高"。

### 7.3 词条组合与协同（Combo）

为奖励"凑对"，定义协同加成：

| 组合 | 条件 | 协同效果 |
|------|------|---------|
| 暴击流 | 暴击率 + 暴击伤害 同件出现 | 触发额外 +8% 暴伤（套用 combo 标记） |
| 元素流 | 同元素伤害 ≥2 条 | 该元素伤害再 +10% |
| 生存流 | 生命 + 双防 同件 | 生命额外 +5% |
| 极速流 | 攻击速度 + 移速 | 攻速再 +3% |

> Combo 仅作展示性增益，不破坏平衡；前端在 `renderEquip` 时高亮协同词条。

### 7.4 套装系统（可选扩展）

- 集齐同套装 N 件触发套装属性（如"烈焰套 3 件：火伤+15%"）。
- 套装件分散在不同 Boss 掉落，延长追求周期（留存）。

### 7.5 去重与互斥规则

- 同一装备上 **不可出现两条同名词条**。
- 互斥对示例：`吸血` 与 `反伤` 不可同件；`纯攻`前缀与`纯防`后缀无互斥（鼓励攻防兼备）。
- 主属性词条（武器攻击）与随机"攻击+"可共存，叠加生效。

> 去重/互斥的实现与校验见 §6.6.5（生成管线冲突检测）与 §6.6.8（边界处理）；升阶（材料/成功率）实现见 §6.6.7。

## 8. 装备附魔与强化系统（部分已实现：强化 +N 与词缀重铸/洗练见 §12.11）

### 8.1 强化（+N）

| 等级 | 效果 | 失败率 | 失败惩罚 | 所需材料 |
|------|------|--------|---------|---------|
| +0~+5 | 主属性 +6%/级 | 0% | 无 | 金币 + 黑铁石 |
| +6~+9 | 主属性 +8%/级 | 10%~25% | 降 1 级 | 金币 + 黑铁石 |
| +10~+12 | 主属性 +10%/级 | 30%~45% | 降 1~2 级 | 金币 + 玄铁石 |
| +13~+15 | 主属性 +12%/级 | 50%~65% | 降 2 级 | 金币 + 神铁石 |

- **保护符**：消耗 `强化保护符` 可使失败不降级（仅损耗材料）。
- 强化加成写入装备实例，随装备被 `calcPlayerStats` 汇总到角色属性。

### 8.2 附魔（附加词条）

- 使用 `附魔卷轴` 在装备 **空词条槽**（高品质装备预留空槽）追加一条随机词条。
- 卷轴分阶：初级（绿词条池）/中级（蓝紫池）/高级（橙红池）。
- 附魔后绑定该装备（防洗白刷词）。

### 8.3 洗练（重 roll）

- 使用 `洗练石` 对 **单条已有随机词条** 重新 roll 数值（保留词条名，刷新数值与 tier）。
- `锁词条` 功能：消耗额外洗练石锁定某条不被动，专注优化其他条。

### 8.4 宝石镶嵌（可选）

- 装备带孔（武器/铠甲默认 1~2 孔），镶嵌 `攻击/防御/生命/暴击 宝石`。
- 宝石可拆可合（3 合 1 升阶），构成次级经济。

### 8.5 传承 / 转移

- `传承符`：将旧装备的强化等级无损转移到新装备（旧装备销毁），解决"换装即白肝"痛点，降低沉没成本焦虑（见 §9.3）。

### 8.6 成本曲线与失败保护（设计意图）

- 强化成本 **指数增长**（金币×1.6^等级），天然限速，避免一天毕业。
- 全程提供保护符/传承符作为"付费或肝度"安全阀，确保非付费玩家也有稳定毕业路径（合规友好）。

## 9. 装备交易与经济系统（部分已实现：经济闭环见 §12.10；拍卖行 UI 待做）

### 9.1 货币体系

| 货币 | 获取 | 用途 | 绑定 |
|------|------|------|------|
| 金币 | 打怪/Boss/任务 | 强化/洗练/基础消费 | 可交易 |

### 9.2 交易方式

- **拍卖行**：玩家挂单（装备/材料），全服竞价或一口价；支持按品质/等级筛选。
- **邮件交易**：点对点邮寄装备+金币（防误操作二次确认）。
- **摆摊**（可选）：主城固定区玩家互刷。

### 9.3 绑定规则

| 规则 | 说明 |
|------|------|
| BoE（装备后绑定） | 未装备前可交易，装备后绑定本人 |
| BoP（拾取绑定） | 神话/Boss 专属，拾取即绑，仅可邮件给同账号角色 |
| 材料 | 大多可交易；高级材料部分绑定 |

### 9.4 经济来源与消耗（Sink/Source）

| 类别 | 来源 Source | 消耗 Sink |
|------|-----------|----------|
| 金币 | 打怪、卖垃圾装、任务 | 强化、洗练、拍卖税、修理 |
| 元宝 | 活动、成就 | 保护符、传承符、外观 |
| 材料 | 副本、Boss、分解 | 强化、附魔、合成 |

### 9.5 定价与税收

- 拍卖行收取 **5% 交易税**（金币形式销毁，抗通胀）。
- 系统基于近期成交价给出 **推荐价区间**，降低信息不对称。
- 高价神话装备设 **公示期 + 全服通告**，兼具炫耀与价格发现。

### 9.6 反通胀与合规

- 金币主要 Sink（强化消耗、税收）设计为 **长期大于来源**，缓慢回收。
- 定期"装备淘汰"（新地图更高 itemLevel）自然稀释旧装价值。
- 遵守国内 H5/小游戏合规：实名、防沉迷时长提醒、无开箱式直接付费抽奖（材料可肝）。

## 10. 玩家心理与留存激励系统（部分已实现：周常/成就见 §12.12；排行榜/活动待做）

### 10.1 Hook 模型应用

| 阶段 | 本作实现 |
|------|---------|
| Trigger 触发 | 每日登录推送、Boss 刷新提醒、好友组队邀请 |
| Action 行动 | 一键进图、自动战斗辅助降低操作门槛 |
| Variable Reward 可变奖励 | **词条随机 roll**、品质概率、Boss 掉宝——不确定奖励最成瘾 |
| Investment 投入 | 强化/附魔沉没成本、社交关系、排行榜名次 |

### 10.2 可变奖励设计（落实 P3）

- 每次掉落都是一次"小赌"：看不到词条前永远有"可能极品"的期待。
- 单条词缀的 `rollTier`（普通/优良/完美）制造"差一点"的追悔与再战欲。
- 神话 0.05% 掉率 + 全服通告，制造"别人能出我也能"的社交驱动。

### 10.3 损失厌恶与沉没成本

- **传承系统**（§7.5）把"换装损失"降到最低，反而鼓励玩家持续投入强化——因为旧投入不浪费。
- 每日登录连续奖励：断签即清空，利用损失厌恶促活。
- 装备绑定（BoE）让玩家对"自己的套装"产生所有权感。

### 10.4 登录与连续奖励

- **7 日循环**：第 1~6 天递增奖励，第 7 天给"周常大礼"（保护符/高级材料）。
- 断签重置，但保留已领天数（温和惩罚）。
- 月度累计登录额外奖励，拉长 LTV。

### 10.5 成就与里程碑

- 成就树：首杀 Boss / 单件 +15 / 集齐套装 / 交易额达标……
- 成就给 **永久称号 + 少量属性**，兼顾荣誉与数值。
- 里程碑（等级/战力阈值）触发全服播报，强化正反馈。

### 10.6 社交与排行榜

- **战力排行榜**（全服/同服）、**Boss 首杀榜**、**富翁榜**。
- 组队 Boss：鼓励公会/好友协作，提升留存与付费社交场景。
- 交易行的"炫耀挂单"本身即社交货币。

### 10.7 活动与 FOMO

- 限时双倍掉落、节日 Boss、赛季排行——制造"现在不玩就亏"的紧迫感。
- 但严控频率，避免过度消耗导致流失。

### 10.8 新手引导（前 10 分钟）

1. 选职业（预览三职业特点）→ 2. 自动进比奇城郊 → 3. 教学怪引导普攻/技能 → 4. 掉第一件绿装引导"对比装备" → 5. 引导强化 +3 → 6. 提示签到与每日任务。

## 12. 技术实现与部署

### 12.1 前端架构

- 单文件 `index.html`（Canvas2D），`account.js` 独立承载账号/技能树/进阶。
- `calcPlayerStats(prof, level, equipment)` 为属性唯一出口，所有加成（装备/技能树/进阶/强化）经 `extraBonuses` 汇入。
- 性能预算（延续技术美术标准）：同屏粒子 ≤ 2000（PC）/ 500（移动）；单图 draw call 受 Canvas 限制以"对象数"计（≤ 400）；Boss 战帧时间 ≤ 预算。

### 12.2 后端架构（零依赖 Node）

- `db.js`：零依赖 `node:sqlite`（Node 22 需 `--experimental-sqlite`，24+ 免标志；`start.js` 按版本自动判断）；建 `users/profiles/saves` 三表，`saves` 以 `slot`（职业名）为 PRIMARY KEY 存 JSON 文本；自动探测不可用则回退 JSON 文件，首启迁移旧 `db.json`。
- `auth.js`：scrypt 哈希口令、Bearer Token 会话（2h / 30天记住登录）、档案白名单字段。
- `mail.js`：无 SMTP 时 dev 回退写 `.mailbox/` + 控制台链接。
- `index.js`：静态托管 + `/api/*` 路由（register/login/logout/me/profile/recover/reset/verify）。

### 12.3 数据互通原则

- **服务端 SQLite 为存档权威源**：`saveGame()` 先写 `localStorage`（即时离线兜底），再异步 `POST /api/save {slot, data}`；`loadGame(slot)` 优先 `GET /api/load?slot=`，失败回退 localStorage（兼容旧单键）。
- **存档按职业分槽**：`SAVE_KEY_OF(prof) = SAVE_KEY + ':' + prof`，战士/法师/道士各自独立存档，切换职业不互相覆盖。
- **持久化范围**：等级/经验/金币/装备(`playerEquipment`)/背包(`playerBackpack`)/技能树/进阶/任务进度 —— 刷新与换端均完整还原。
- `extraBonuses` 全局对象保证"前端即时反馈、后端持久真相"双层一致。
- 相关 API：`POST /api/save`、`GET /api/load`、`GET /api/saves`、`DELETE /api/save`。

### 12.4 小程序 / 小游戏部署

- 当前原型为纯 Canvas 程序化绘制（无外部图片），对微信小游戏转制友好。
- 路径：方案 B（转微信小游戏原生 Canvas 运行，性能优于 web-view 嵌入）。
- 账号体系复用同域 Node 后端（小游戏内 `wx.request` 调 API）。

### 12.6 背包系统与物品模型（已落地）

**物品三大类（`kind`）**

| kind | 来源 | 堆叠 | 数据键 | 示例 |
|------|------|------|--------|------|
| `equip` 装备 | 怪物/Boss 掉落 `drops[].equip` | 不堆叠，每件独立 | `slotKey/quality/affixes` | 武器/衣服/戒指… |
| `consumable` 消耗品 | 怪物死亡 30% 掉 `drops[].item` | 按 `id` 堆叠，上限 20 | `CONSUMABLE_DEFS` | 生命药水/魔法药水/大生命药水/回城卷轴 |
| `material` 材料 | 怪物死亡按地图阶掉 `drops[].item` | 按 `id` 堆叠，上限 20 | `MATERIAL_DEFS` | 铁矿石/草药/魔物核心/龙鳞 |

**背包 UI（已实装 `renderBackpack`）**
- 容量上限 **20 格**；标签页 **5 类**：全部 / 武器 / 防具 / 消耗品 / 材料（仅显示非空分类，带计数）。
- 装备：显示图标+名称+品质色+词条数，点击弹对比面板（与已穿同槽对比）；可"装备/丢弃"。
- 消耗品/材料：显示 `bp-item-icon`+名称+`×qty`+描述；消耗品可"使用"（回血/回蓝/回城）。
- 原生 HTML5 拖拽（`draggable`/`ondragstart`/`ondragover`/`ondrop`）：背包内拖拽排序；背包↔装备栏双向；装备栏拖回背包 = 卸下（`unequipToBackpack`）。

**装备栏（10 格 `SLOT_DEFS`）** —— 见 §6.1。

**使用与持久化**
- `useConsumable(uid)`：生命药水回 35% 最大 HP、大生命药水回 70%、魔法药水回 40% MP、回城卷轴返回比奇城郊。
- 全部变更经 `saveGame()` 落盘（localStorage + 服务端 SQLite），刷新后背包/穿戴自动恢复（见 §12.3）。
- 验收：puppeteer 端到端 **15/15** 通过，装备/消耗品/材料刷新后均从服务端 SQLite 还原，无 pageerror / console error。

### 12.7 音频系统（已落地）

零外部音频资源，全程 `WebAudio` 程序化合成（`Sound` 模块，`index.html` ~L7959）。

- **引擎**：`AudioContext` 懒初始化（首次用户手势内 `resume()`，合规绕过浏览器自动播放限制）；`master` 增益总线，默认音量 0.45。
- **音效清单**：

  | 事件 | 方法 | 音色 / 合成 |
  |------|------|-------------|
  | 命中 | `Sound.hit()` | 方波 200→110Hz + 噪声，55ms 节流防吵 |
  | 受击 | `Sound.hurt()` | 锯齿 150→70Hz + 低频噪声 |
  | 技能释放 | `Sound.cast(effect)` | 按 `effect` 分音色：fireSwirl / iceBlast / lightning / heal(上行三音) / shield / charge / poison / 默认 slash |
  | 击杀 | `Sound.enemyDie()` | 方波 320→90Hz + 噪声 |
  | 升级 | `Sound.levelup()` | 523/659/784/1047 上行琶音 |
  | 掉落（分档） | `Sound.drop(q)` | 按品质 0普通..5神话：普通单音"叮"→传说"嗡"→神话全奏+低音（对应 §6.5 品质分档音效） |
  | Boss 出现 | `Sound.bossAppear()` | 低沉长音预警 |
  | 玩家死亡 | `Sound.gameover()` | 下行哀音 |
  | UI 点击 | `Sound.click()` | 短促高频 |

- **静音开关**：HUD 🔊/🔇 切换，`Sound.toggle()` 持久化 `localStorage['legends_muted']`，刷新保留偏好。
- **验收**：真机零音频报错；静音态下所有方法短路，不创建音频节点。

## 13. 合规与风险控制

- **实名 + 防沉迷**：未成年时长/消费限制，到点提醒。
- **无付费抽奖**：核心随机性来自玩法掉落（可肝），不卖"开箱钥匙"。
- **透明概率**：品质掉率、强化失败率全部在游戏内公示。
- **数据安全**：口令 scrypt 加盐哈希，Token 随机 UUID，会话过期可控。
- **内容合规**：无暴力血腥过度描写，Boss 死亡用碎裂粒子而非写实血腥。
